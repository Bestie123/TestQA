# Правила оформления тест-кейсов и баг-репортов (Zephyr Scale + Jira)

## Общие правила

- Все ответы на русском языке
- При создании тест-кейса или баг-репорта Я ОБЯЗАН задавать уточняющие вопросы, если в запросе недостаточно данных
- Формат выдачи: `.docx` (Word) через HTML → Word COM конвертацию
- После генерации файл автоматически открывается через `Start-Process`

## Что я должен уточнять перед созданием

Если пользователь запросил тест-кейс или баг-репорт, но не предоставил всех данных, я задаю уточняющие вопросы:

### Для тест-кейса:
1. **Название тест-кейса** (title) — как он должен называться?
2. **Модуль/раздел** — к какому модулю относится?
3. **Приоритет** — High / Medium / Low?
4. **Тип** — Positive / Negative / Edge?
5. **Предусловия** — какие шаги нужно выполнить перед тестом?
6. **Шаги воспроизведения** — последовательность действий (пошагово)
7. **Ожидаемый результат** — что должно произойти после выполнения шагов?
8. **Ссылки на связанные задачи** — IBPA / Zephyr ключи?

### Для баг-репорта:
1. **Предварительные условия** — что должно быть настроено/создано?
2. **STR (Шаги воспроизведения)** — пошаговая инструкция
3. **ER (Ожидаемый результат)** — как должно работать?
4. **AR (Фактический результат)** — что происходит на самом деле?
5. **Окружение** — среда, браузер, версия?

## Формат тест-кейса (Zephyr Scale)

### Структура:
1. **Preconditions (Предусловия)** — нумерованный список `<ol>`
2. **Steps (Шаги)** — нумерованный список `<ol>` с действиями
3. **Expected Result (Ожидаемый результат)** — один или несколько пунктов

### Форматирование текста (Froala Editor v4.1.4):
| Элемент | Тег HTML | Пример |
|---------|----------|--------|
| UI-элементы (кнопки, поля, вкладки, роли) | `<b>` | `<b>На согласование</b>` |
| URL / пути | `<code>` внутри `<a>` | `<a href="..."><code>https://...</code></a>` |
| Кавычки вокруг UI-элементов | `&quot;<b>...</b>&quot;` | `&quot;<b>Инициация</b>&quot;` |
| Ссылки на тест-кейсы | `<a href="...">KEY</a>` | `<a href="...">IBPA-T1418</a>` |
| Ссылки на Jira issues | `<a href="...">KEY</a>` | `<a href="...">IBPA-4614</a>` |
| Ссылки на тестовую среду | `<a href="..."><code>URL</code></a>` | полный URL как текст |
| Стрелка между действиями | ` → ` (текст) | |
| Уточнения в скобках | обычный текст (не жирный) | |

### Формат ссылок:
- **Тест-кейсы Zephyr**: `<a href="https://jira.ifellow.ru/secure/Tests.jspa#/testCase/KEY?projectId=10904">KEY</a>`
- **Jira issues**: `<a href="https://devjira.ifellow.ru/browse/KEY">KEY</a>`
- **Тестовая среда**: `<a href="https://devjira.ifellow.ru/"><code>https://devjira.ifellow.ru/</code></a>`

### Запрещено:
- Разные шрифты в одном поле
- Избыточное использование цвета
- Вложенность списков глубже 2 уровней
- Таблицы внутри списков
- Подчеркивание не-ссылок
- URL без гиперссылки (все URL должны быть кликабельными)

## Формат баг-репорта (Bug Report)

### Структура (4 обязательные секции):

**1. Предварительные условия** — `<b>Предварительные условия:</b>` + `<ol>`
- Форматирование как в тест-кейсе
- Кавычки: `&quot;<b>Команда пресейл проекта</b>&quot;`

**2. STR (Шаги воспроизведения)** — `<b>STR (Шаги воспроизведения):</b>` + `<ol>`
- Каждый шаг — один пункт списка
- UI-элементы в `<b>`
- Кавычки: `Нажать &quot;<b>На согласование</b>&quot;`
- Стрелка между последовательными действиями: ` → `
- Уточнения в круглых скобках

**3. ER (Ожидаемый результат)** — `<b>ER (Ожидаемый результат):</b>` + `<ul>` или `<ol>`
- Статус доступности: `<b>недоступно</b>` / `<b>доступно</b>`

**4. AR (Фактический результат)** — `<b>AR (Фактический результат):</b>` + `<ul>` или `<ol>`
- Структура идентична ER, описывает ошибочное поведение

## Обязательные поля для баг-репорта (вывод в чат перед генерацией)

При создании баг-репорта я ОБЯЗАН вывести в чат следующие поля и дождаться подтверждения/корректировки от пользователя:

### Критерии определения

**Главное правило — баг определяется разницей между ER и AR.** Именно различие между ожидаемым и фактическим результатом указывает на суть дефекта. Если функционал работает (например, валидация блокирует отправку), но сообщение некорректно — это баг именно в сообщении, а не в валидации.

**Варианты значений:**

- **Приоритет:** highest / критический / обычный / lowest
- **Степень влияния:** высокая / средняя / низкая
- **Фаза выявления:** ST / IFT / prod incident
- **Серьезность:** s0 — тривиальный / s1 — незначительный / s2 — значительный / s3 — критический / s4 — блокирующий
- **Функциональность:** (выбрать из списка ниже)
  1. Настройки Jira-проекта — общие настройки, схемы прав доступа, доски, Service Desk, SLA
  2. Жизненные циклы — статусы, переходы, валидаторы, условия
  3. Задачи — типы задач, связи задач
  4. Экраны и поля — экраны, поля (атрибутивный состав полей, настройка отображения, поведение полей на экране — НЕ скриптовая логика)
  5. Пользователи и группы — пользователи, группы
  6. Дашборды — дашборды
  7. Структуры — структуры
  8. Планы — планы и программы Advanced Roadmap
  9. Insight — объектные схемы, типы объектов, объекты, автоматизации
  10. Скрипты — постфункции, джобы, листенеры, бихейвиоры, кондишны, скриптовые поля
  11. Веб-фрагменты — веб-фрагменты создания задач, отображения таблиц
  12. Уведомления — уведомления
  13. Интеграции — интеграции
  14. DWH — бэк DWH
  15. ETL — ETL DWH
  16. Отчеты — отчеты BI
  17. Прочее — если ничего из вышеописанного не подошло
  18. Мульти — если подходит несколько значений
- **Вид дефекта:** функциональный / визуальный / логический / архитектурный / инфраструктурный / дефект UX / дефект нагрузки
- **Среда выявления:** test/dev / stage / prod

### Примеры оценки

1. **Валидация срабатывает, но сообщение на английском вместо русского** → ошибка именно в тексте сообщения, а не в логике. Вид: дефект UX. Серьезность: s1. Приоритет: обычный. Функциональность: зависит от того, что не так — если поле на экране → Экраны и поля.
2. **Поле доступно для редактирования, хотя должно быть недоступно** → ошибка в правах/условиях. Серьезность: s3. Вид: функциональный. Функциональность: зависит от реализации (скрипты или конфигурация экрана).
3. **Кнопка не срабатывает (ничего не происходит)** → функциональный дефект. Серьезность: s3/s4 в зависимости от блокировки функционала.

Формат вывода в чат:
```markdown
**Приоритет:** обычный
**Степень влияния:** низкая
**Фаза выявления:** ST
**Серьезность:** s1 — незначительный
**Функциональность:** 4. Экраны и поля
**Вид дефекта:** дефект UX
**Среда выявления:** test/dev
```

## Структура тест-кейсов (Zephyr Scale Excel Export)

При анализе существующих тест-кейсов из Excel-выгрузки Zephyr Scale используй следующие правила:

### 18 колонок Excel-выгрузки:
| # | Колонка | Описание |
|---|---------|----------|
| 1 | **Key** | Уникальный ключ тест-кейса (например, `IBPA-T121`) |
| 2 | **Name** | Название тест-кейса |
| 3 | **Status** | `Approved` или `Deprecated` |
| 4 | **Precondition** | Предусловия (HTML/rich text, может содержать ссылки на Jira issue, Zephyr TC) |
| 5 | **Objective** | Цель тест-кейса |
| 6 | **Folder** | Путь в дереве папок Zephyr (например, `/Проекты/Раздел "Согласование в МК (FIN)"/Раздел исполнители/Карточка задачи/Карточка задачи`) |
| 7 | **Priority** | `Normal` (94%), `High` (3%), `Low` (3%) |
| 8 | **Component** | Компонент |
| 9 | **Labels** | Метки |
| 10 | **Owner** | Владелец (например, `JIRAUSER18603`) |
| 11 | **Estimated Time** | Оценка времени |
| 12 | **Coverage (Issues)** | Связанные Jira issues (например, `IBPA-1814`, `IBPA-1968`, `IBPA-2946`) |
| 13 | **Coverage (Pages)** | Связанные Wiki-страницы |
| 14 | **Test Script (Step-by-Step) - Step** | **Шаг теста** — описание действия |
| 15 | **Test Script (Step-by-Step) - Test Data** | **Тестовые данные** (может быть пустым) |
| 16 | **Test Script (Step-by-Step) - Expected Result** | **Ожидаемый результат** шага |
| 17 | **Test Script (Plain Text)** | Альтернативное текстовое описание |
| 18 | **Test Script (BDD)** | BDD-сценарий (пусто, не используется) |

### Multi-row формат (критически важно):
Один тест-кейс занимает **несколько строк** в Excel:
- **Первая строка** тест-кейса содержит **Key** (колонка 1) + все метаданные (Name, Precondition, Objective, Folder, Priority, Status, Owner, Coverage)
- **Последующие строки** (без Key, колонка 1 пуста) содержат **Step** (колонка 14), **Test Data** (колонка 15), **Expected Result** (колонка 16) — каждый шаг на отдельной строке
- Количество строк с шагами = количество шагов тест-кейса

## Процесс генерации .docx

1. Подготовить HTML с полным форматированием
2. Сохранить HTML во временный файл `$env:TEMP\precondition.html`
3. Убить процессы WINWORD: `Get-Process -Name WINWORD | Stop-Process -Force`
4. Выждать 2-3 сек
5. Создать Word COM, открыть HTML, сохранить как DOCX (wdFormatDocumentDefault = 16)
6. Закрыть Word
7. Скопировать файл в `precondition.docx` (или `bugreport.docx`)
8. Удалить временные файлы
9. Открыть через `Start-Process`
